# Obsidian Notes Backend

## Recommended Setup

Use a private GitHub repo:

```text
jbellsolutions/speakeragent-ops-vault
```

Open that repo as an Obsidian vault locally.

Configure Railway:

```bash
OBSIDIAN_GITHUB_REPO=jbellsolutions/speakeragent-ops-vault
OBSIDIAN_GITHUB_BRANCH=main
OBSIDIAN_VAULT_PATH=SpeakerAgent Ops
```

## Note Structure

The service writes:

```text
SpeakerAgent Ops/
  00 Inbox/
  10 Daily Reports/
  20 Runtime Incidents/
  30 Architecture/
  40 Skills And Workflows/
  YYYY-MM-DD/
    HHMMSS-runtime-report.md
    HHMMSS-daily-engineering-report.md
```

The dated folders are generated by the Railway service. The numbered folders are stable places for Lester and Codex to keep curated infrastructure notes, architecture decisions, runbooks, skills, and workflow improvements.

## Why GitHub-Backed Notes

Railway local disk is ephemeral. GitHub gives durable notes that can be opened in Obsidian and reviewed over time.

## Dry Run

When `DRY_RUN=true`, notes are written only to local `vault/` inside the container. That is useful for testing but not durable.

## Starter Vault Files

Recommended starter files for the private vault:

```text
README.md
SpeakerAgent Ops/00 Inbox/README.md
SpeakerAgent Ops/10 Daily Reports/README.md
SpeakerAgent Ops/20 Runtime Incidents/README.md
SpeakerAgent Ops/30 Architecture/README.md
SpeakerAgent Ops/40 Skills And Workflows/README.md
```

The ops agent never writes secrets to these notes. Keep API keys, tokens, webhook URLs, and production credentials only in Railway environment variables.
